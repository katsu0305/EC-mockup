# ローカルモック生成ツール構想

## 1. 目的

関連仕様:

- .env の正式キー一覧は env仕様書.md を参照
- Live Site Inspector の CLI 契約は CLI仕様_LiveSiteInspector.md を参照
- フェーズ別の実装手順は 実装手順書_一覧.md を参照

GitHub 上で管理されている EC サイトの HTML / CSS / JS テンプレート群と、
EC-database-tang から取得できる製品データを元に、
ローカルで閲覧可能な静的 Web サイトパッケージを生成する。

初期実装は CLI ツールとして開発し、
その後に PyInstaller を使って GUI アプリとして配布できる形も視野に入れる。

このツールの目的は、以下をブラウザ上で確認できるようにすること。

- どのような見た目のサイトか
- どのような商品が掲載されているか
- 商品価格がどの程度で表示されるか
- カテゴリーや商品詳細への遷移がどのように見えるか

一方で、以下の機能は再現対象外とする。

- 商品注文
- 会員登録
- ログイン
- 問い合わせ
- お気に入り
- レビュー投稿
- 検索の完全再現

これらはリンク切れ、ダミーリンク、または未対応ページ表示でよい。

---

## 2. 前提の変更点

当初はローカルパスにあるテンプレート群を直接入力にする想定だったが、
今後は HTML / CSS / JS のソースを GitHub から取得する前提に変更する。

### 入力元

1. GitHub リポジトリ
	 - EC サイトのテンプレートソース
	 - 必要な静的アセット
	 - お知らせテンプレートや補助ファイル

2. EC-database-tang
	 - 商品情報
	 - カテゴリー情報の復元に必要な情報
	 - 商品説明、価格、在庫、画像 URL など

3. MakeShop 画像用 FTP / 公開 URL
	- 商品画像や運用画像の取得元
	- FTP 上のファイル一覧取得
	- 公開 URL からのダウンロード

4. 実ショップサイト
	- 実際に配信されている HTML / CSS / JS の取得元
	- 見た目確認、DOM 構造確認、差分確認のための補助入力
	- 会員制クローズドサイトへのログイン後スナップショット取得対象

このため、ツールは「テンプレート取得」「データ取得」「画像取得」を分離し、必要に応じて「実ショップ参照」を補助的に扱う設計とする。

---

## 3. ゴールイメージ

ツール実行後、以下のようなローカル閲覧用成果物が生成される状態を目指す。

- index.html
- category ごとの一覧ページ
- product ごとの詳細ページ
- お知らせ一覧 / 詳細ページ
- CSS / JS / 画像などを含む assets 一式

生成された成果物は、ローカルの静的サーバでそのまま確認できるようにする。

例:

- output/mock-site/index.html
- output/mock-site/category/pc-accessory/index.html
- output/mock-site/products/BM-MNLBP01NBK/index.html

## 4. ツールの基本方針

### 4-1. MakeShop を完全再現しない

MakeShop 独自テンプレート構文や会員状態、カート挙動を完全再現するのではなく、
閲覧用モックとして必要なページだけを静的 HTML に変換する。

### 4-2. GitHub をテンプレートの正本にする

ローカルに置かれた original フォルダを直接参照するのではなく、
実行時に GitHub から対象 ref のテンプレート一式を取得して作業用ディレクトリへ展開する。

実ショップから取得した HTML / CSS / JS は正本にはせず、以下の補助用途に限定して使う。

- 現在の本番見た目との比較
- テンプレートと実出力の差分確認
- 不足 CSS / JS / DOM 構造の確認
- レンダラ調整時の参照資料

取得対象は HTML だけではなく、画面表示に必要な CSS と JS も含めたスナップショットとする。

### 4-3. 商品データは正規化して扱う

EC-database-tang の API や DB モデルをそのまま画面に流し込まず、
モック生成専用の内部データモデルへ変換してから使う。

### 4-4. ビルダーは取得元を意識しない

HTML 生成処理は GitHub や DB に直接依存せず、
取得済みテンプレートと正規化済み JSON を入力として動く構造にする。

### 4-5. CLI をコアにして GUI は外側に載せる

最初から GUI 主体で作るのではなく、
ビルド処理の本体は CLI と Python モジュールとして実装する。

GUI 化する場合は、その CLI / モジュールを呼び出す薄い画面層を追加する。

これにより、以下を両立しやすくする。

- 開発初期の実装速度
- 自動テストのしやすさ
- GUI 化後の保守性
- PyInstaller での単体配布

### 4-6. 画像は参照されたものだけ取得する

FTP 領域や公開画像を全件ミラーするのではなく、
テンプレート内または商品データ内で参照された画像だけを収集対象にする。

これにより、生成時間とローカル出力サイズを抑えつつ、
モック閲覧に必要な見た目を優先できるようにする。

## 5. 想定アーキテクチャ

### 5-1. 全体構成

1. CLI / Application Service
	 - コマンド実行入口
	 - GUI からも再利用される実行ユースケース

2. Source Fetcher
	 - GitHub からテンプレートソースを取得する

3. Data Exporter
	 - EC-database-tang から商品データを取得して JSON 化する

4. Image Fetcher
	- FTP または公開 URL から必要画像を取得する

5. Live Site Inspector
	- 実ショップの HTML / CSS / JS を取得して参照する
	- DOM 構造、適用 CSS、補助アセットの確認に利用する
	- クローズドサイト用の認証情報を .env から読み込む

6. Normalizer
	 - テンプレートと商品データをモック生成向けの内部形式へ変換する

7. Static Builder
	 - トップ、カテゴリ、商品詳細、お知らせ等の静的 HTML を生成する

8. Preview Server
	 - 生成物をローカルで確認できるようにする

9. GUI Shell
	 - 設定入力画面
	 - 実行ボタン
	 - 進捗表示
	 - ログ表示
	 - 出力先を開く導線

### 5-2. 処理フロー

1. 設定ファイルを読み込む
2. GitHub から対象リポジトリ / ブランチ / タグ / コミットを取得する
3. 作業ディレクトリにテンプレートスナップショットを展開する
4. EC-database-tang から商品データを取得する
5. 必要に応じて実ショップの HTML / CSS / JS を取得し、差分確認用情報として保存する
6. テンプレートと商品データから必要画像の参照先を抽出する
7. FTP または公開 URL から必要画像を取得する
8. 商品データをモック用の内部モデルへ正規化する
9. テンプレートの必要部分を静的 HTML として生成する
10. assets をコピーしてリンクを調整する
11. 必要に応じて実ショップとの差分を確認する
12. output 配下にローカル閲覧用サイトを出力する

## 6. GitHub ソース取得の方針

### 6-1. 優先方式

最初の実装では、git clone または sparse-checkout を優先する。

理由:

- ブランチ指定、タグ指定、コミット指定がしやすい
- テンプレート群をまとめて取得しやすい
- 再現性を持たせやすい

### 6-2. 取得対象

最低限、以下のようなパスを対象にする。

- original/
- assets/
- お知らせ/

必要に応じて将来的に対象パスを追加可能にする。

### 6-3. 認証

private repository を想定し、GitHub Token を環境変数で受け取れるようにする。

例:

- GITHUB_TOKEN

### 6-4. 再現性

ブランチ名だけでなく、可能なら commit SHA 指定に対応する。
これにより「この時点のソースを使ったモック」を再生成できるようにする。

### 6-5. 実ショップ参照の位置づけ

実ショップの HTML / CSS / JS は、GitHub テンプレートで不足する情報を補うための参照資料として扱う。

ただし、以下の扱いに留める。

- スクレイピング結果を生成の正本にしない
- 実ショップ取得物は比較用スナップショットとして保存する
- 会員状態や動的描画結果に依存する内容は、そのまま再現対象にしない

アクセスに必要な URL、ID、パスワード、必要なら追加ヘッダーや cookie 情報は、ローカルの .env から読み込む。

## 7. 実ショップ参照の方針

### 7-1. 利用目的

実ショップ参照は、主に以下のために使う。

- DOM 構造の確認
- 画面ごとの見た目確認
- CSS 適用状況の確認
- GitHub テンプレートとの差分把握
- ローカル生成物とのズレ確認

### 7-2. 取得方式

実ショップ参照は、対象ページの HTML のみではなく、関連する CSS / JS も合わせて取得する方式とする。

初期方針:

- ログイン済み状態で対象ページへアクセスする
- 表示後の HTML を保存する
- 参照されている CSS を保存する
- 参照されている JS を保存する
- 必要に応じて画像 URL も参照情報として控える

クローズドサイトのため、アクセス情報は .env に記載した値を使用する。

想定する .env 項目例:

- LIVE_SITE_BASE_URL
- LIVE_SITE_LOGIN_URL
- LIVE_SITE_USERNAME
- LIVE_SITE_PASSWORD
- LIVE_SITE_LOGIN_USER_FIELD
- LIVE_SITE_LOGIN_PASSWORD_FIELD
- LIVE_SITE_LOGIN_SUBMIT_SELECTOR

必要であれば追加で以下も扱えるようにする。

- LIVE_SITE_COOKIE
- LIVE_SITE_USER_AGENT
- LIVE_SITE_EXTRA_HEADERS

### 7-3. 非採用方針

以下は採用しない。

- 実ショップ HTML をそのまま主テンプレートとして採用すること
- 実ショップから全画面をクロールして生成元にすること
- 会員限定状態の DOM をそのまま仕様化すること

---

## 8. 商品データ取得の方針

### 7-1. データ取得方法

初期実装では以下のいずれかで商品データを取得する。

1. EC-database-tang の API を利用する
2. EC-database-tang の DB / モデルを直接読む

まずは API 利用を第一候補とする。

理由:

- 取得責務を分離しやすい
- モック生成ツール側が DB 実装に強く依存しない
- JSON として扱いやすい

### 7-2. 必要なデータ項目

最低限必要なのは以下。

- system_code
- custom_code
- product_name
- sell_price
- consumer_price
- quantity
- pc_content
- product_group_code
- product_group_name
- 商品画像 URL
- カテゴリー情報
- バリエーション情報

### 7-3. 内部モデル化

商品データは、そのままテンプレートへ流し込まず、
以下のような ViewModel に変換する。

- SiteContext
- ProductCard
- ProductDetail
- CategorySummary
- NewsItem

## 8. 画像取得の方針

### 8-1. 画像取得元

画像は以下の優先順で扱う。

1. GitHub 管理下の画像アセット
2. EC-database-tang が返す商品画像 URL
3. MakeShop の公開画像 URL
4. MakeShop の FTP 領域

### 8-2. FTP の役割

FTP は画像の一覧取得や未公開ファイルの回収経路として扱う。
ただし、ブラウザ表示用に直接使う URL は、可能であれば公開 URL に正規化する。

つまり、取得時は FTP、出力 HTML 上の参照先はローカル配置された画像パス、という形にする。

### 8-3. 画像収集戦略

初期実装では、以下の 2 系統から必要画像候補を集める。

1. テンプレート内の img src、CSS background-image、script 内の画像参照
2. 商品データ内の画像 URL

その上で、URL またはファイル名の重複を排除し、未取得の画像だけをダウンロードする。

### 8-4. ローカル保存方針

取得した画像は output 配下へ再配置する。

例:

- output/mock-site/assets/images/products/
- output/mock-site/assets/images/banners/
- output/mock-site/assets/images/content/

HTML 内の参照は、最終的にローカルパスへ置き換える。

### 8-5. 容量制御

画像は全件取得ではなく参照ベース取得とし、必要に応じて以下の制御を入れる。

- 最大ダウンロード件数
- 総容量上限
- 拡張子フィルタ
- タイムアウトとリトライ

### 8-6. 認証情報の扱い

FTP 接続情報は構想書や設定ファイルに平文で固定しない。
接続先ホスト、ユーザー名、パスワードは環境変数で受け取る。

例:

- MAKESHOP_FTP_HOST
- MAKESHOP_FTP_USER
- MAKESHOP_FTP_PASSWORD
- MAKESHOP_PUBLIC_BASE_URL

## 12. 設定ファイル案

```yaml
source:
	github:
		repository: your-org/ECsite
		ref: main
		subpaths:
			- original
			- assets
			- お知らせ
		auth:
			type: token-env
			env_name: GITHUB_TOKEN

data:
	ec_database_tang:
		mode: api
		base_url: http://127.0.0.1:8000/api/v1
		api_key_env: ECDB_API_KEY
		latest_only: true

images:
	mode: download
	prefer_public_url: true
	public_base_url_env: MAKESHOP_PUBLIC_BASE_URL
	ftp:
		host_env: MAKESHOP_FTP_HOST
		user_env: MAKESHOP_FTP_USER
		password_env: MAKESHOP_FTP_PASSWORD
	include_patterns:
		- .jpg
		- .jpeg
		- .png
		- .gif
	max_files: 500
	total_size_limit_mb: 300

build:
	output_dir: ./output/mock-site
	temp_dir: ./work
	pages:
		- top
		- category
		- product
		- news
	disable_actions:
		- cart
		- member_entry
		- login
		- inquiry

app:
	mode: cli
	gui:
		enabled: false
		remember_last_values: true
```

## 15. GUI 化の方針

### 15-1. GUI の役割

GUI はビルドロジックの本体ではなく、実行操作のフロントエンドとして扱う。

最低限ほしい機能は以下。

- GitHub リポジトリ / ref / 対象パスの入力
- API 接続先や API キー参照名の入力
- FTP 接続情報の参照名入力
- 出力先ディレクトリの指定
- build 実行
- 実行ログ表示
- 完了後に output を開く

### 15-2. 技術選定の考え方

GUI は Python 単体で完結し、PyInstaller と相性のよい構成を優先する。

候補:

- Tkinter
- PySide6

初期は実装と配布の軽さを重視して Tkinter、
画面要件が増えるなら PySide6 を検討する。

### 15-3. PyInstaller 前提の注意点

PyInstaller を使う場合、以下を初期設計から意識する。

- テンプレートや設定雛形などの同梱ファイル管理
- 実行時の作業ディレクトリの扱い
- ログ出力先の固定
- 環境変数が未設定のときの入力導線
- 外部コマンド依存の最小化

特に git 実行や FTP 接続まわりは、GUI から呼んでも同じ結果になるよう、
CLI と共通のサービス層へ閉じ込める。

---

## 16. この構想で重要な判断

このツールは、EC サイト本番の完全再現を目指すものではなく、
確認用の静的モック生成ツールとして割り切る。

そのため、重要なのは以下。

- 見た目が近いこと
- 商品一覧と詳細が見えること
- 価格が確認できること
- 画像がローカルでも崩れず見えること
- ローカルですぐ開けること
- GitHub 上の特定時点のソースで再生成できること
- CLI でも GUI でも同じビルド結果になること

逆に、以下は初期段階で無理に対応しない。

- カートの動作再現
- 会員状態の完全再現
- MakeShop 独自構文の完全解釈
- 実サイトと同等のバックエンド動作

また、GUI 化は初期実装の前提ではあるが、
ビルドロジックより先に画面を作らないことを重要な判断とする。

## 17. まとめ

本ツールは、GitHub 上の EC サイトテンプレートと、EC-database-tang 由来の商品データを使って、
必要画像も取得しながら、ローカル閲覧用の静的モックサイトを生成するためのビルダーである。

開発はまず CLI で進め、ビルド処理を安定させた後に、
同じ処理系を PyInstaller で配布可能な GUI アプリへ展開する。

アーキテクチャ上は、以下の責務を分離する。

- GitHub からテンプレートを取得する責務
